────────────────────────────────────────────────────────────────────────────────
1. OVERVIEW
────────────────────────────────────────────────────────────────────────────────
Siren.ng is a real-time community emergency coordination platform for Lagos,
Nigeria. Citizens report emergencies via WhatsApp or the web app. The platform
verifies incidents with AI, dispatches nearby volunteer responders and registered
organisations, broadcasts community alerts to location subscribers, coordinates
resource needs, and accepts micro-donations — all without requiring app installs
or user accounts.

2. TECH STACK
────────────────────────────────────────────────────────────────────────────────
Backend
  • Django 5.1 (Python 3.13) + Django REST Framework
  • Django Channels 4 (ASGI WebSocket, channels_redis)
  • Celery (async tasks, Redis broker)
  • PostgreSQL via Supabase  (engine: django.db.backends.postgresql)
  • Redis via Railway  (broker + channel layers + cache)
  • Twilio — WhatsApp Business API
  • Anthropic claude-sonnet-4-6 — AI incident verification
  • Paystack — donations + sub-account splits
  • Cloudinary — media upload/storage
  • WhiteNoise — static file serving in production

Frontend
  • React 19 + Vite 6 + TypeScript
  • react-leaflet@4 + Leaflet  (OpenStreetMap, no API key)
  • Zustand  (global state: auth + incidents)
  • TanStack React Query  (data fetching + cache)
  • Recharts  (analytics charts)
  • Tailwind CSS 3.4

Hosting (target)
  • Railway — Django ASGI + Celery worker + Redis
  • Supabase — PostgreSQL
  • Cloudinary — media files

PERMANENT CONSTRAINTS (never change these):
  ✗ NO GeoDjango, NO PointField, NO GDAL, NO postgis engine
  ✓ Location = two FloatFields: location_lat + location_lng
  ✓ Haversine distance via raw SQL in utils/distance.py

5. RUNNING LOCALLY
────────────────────────────────────────────────────────────────────────────────
Prerequisites: Python 3.13 venv activated, Node 20+, Redis accessible.

  # Django API server
  .venv/Scripts/python.exe manage.py migrate
  .venv/Scripts/python.exe manage.py createsuperuser
  .venv/Scripts/python.exe manage.py runserver

  # Celery worker (separate terminal)
  .venv/Scripts/celery -A sirenapp worker -l info

  # Frontend dev server (separate terminal, hot-reload)
  cd frontend
  npm install --legacy-peer-deps
  npm run dev                         # Vite on http://localhost:5173

  # Production build (then Django serves via WhiteNoise)
  cd frontend && npm run build        # outputs to frontend/dist/
  .venv/Scripts/python.exe manage.py collectstatic

Note: Always use .venv/Scripts/python.exe — not bare `python` or `python3`.

6. CORE DATA MODELS

Incident  (apps/incidents/models.py)

  incident_type        FIRE|FLOOD|COLLAPSE|RTA|EXPLOSION|DROWNING|HAZARD|OTHER

  fraud_score          Float 0–1
  ai_raw_response      JSONField
  vouch_count          Int
  vouch_threshold      Int
  total_donations_kobo BigInt
  donation_count       Int
  created_at / updated_at / resolved_at

ResponseLog  (apps/incidents/models.py)
  Immutable audit log of every status change with actor + note.

VouchRecord  (apps/incidents/models.py)
  One vouch per session_hash. Stores distance for fraud detection.

Responder  (apps/responders/models.py)
  phone_hash, skill_category, home_lat/lng, response_radius_km,
  is_available, responds_to (JSONField), paystack_subaccount.

ResponderDispatch  (apps/responders/models.py)
  Through table Responder ↔ Incident.
  Tracks: accepted (nullable bool), on_scene_at, completed_at.
  unique_together = (responder, incident)

Organisation  (apps/organisations/models.py)
  NGO/hospital/fire station: location_lat/lng, org_type, status=VERIFIED,
  responds_to (JSONField), contact_whatsapp, paystack_subaccount.

ResourceItem  (apps/resources/models.py)
  FK→Incident. category, label, status: NEEDED→CLAIMED→ARRIVED.

ResourceClaim  (apps/resources/models.py)
  FK→ResourceItem. unique_together (resource, claimer_hash).
  Duplicate claim returns HTTP 200 (not 400) per spec.

Donation  (apps/resources/models.py)
  FK→Incident. amount_kobo, fund_choice (VICTIM|RESPONDER|PLATFORM),
  paystack_reference (unique), status (PENDING|SUCCESS|FAILED).
  property amount_naira = amount_kobo / 100

LocationSubscription  (apps/subscriptions/models.py)
  phone_hash, whatsapp_number, location_lat/lng, alert_radius_km,
  incident_types (JSONField), is_active.

SubscriptionAlert  (apps/subscriptions/models.py)
  Dedup: unique_together (subscription, incident).

8. WEBSOCKET
────────────────────────────────────────────────────────────────────────────────
URL:    ws://<host>/ws/incidents/
Group:  "incidents"  (broadcast to all connected clients)

Server → client message:
  { "type": "incident.update", "incident": { ...serialized Incident } }

Triggered on: new incident, status change, vouch added.
Frontend hook: useWebSocket.ts — auto-reconnects after 3s on close.
Store: incidentStore.updateIncident() merges update by id.

9. WHATSAPP FLOWS  (apps/whatsapp/)
────────────────────────────────────────────────────────────────────────────────
Inbound routing (handlers.py → route_inbound):

  Keyword / context          → Action
  ──────────────────────────────────────────────────────────────────
  YES / NO                   → Responder accepts/declines dispatch
  ONSCENE / DONE             → Responder marks on-scene / completed
  ACCEPT / DECLINE / CALL    → Organisation replies to notification
  JOIN RESPONDERS            → Responder onboarding (DB registration)
  JOIN ORG                   → Organisation registration prompt
  WATCH                      → Start location subscription flow
  MY ALERTS                  → List user's active subscriptions
  STOP <label>               → Deactivate named subscription
  VOUCH <id>                 → Vouch for incident
  Location share             → Processed in active flow context
  Anything else              → Create new incident report

Multi-step WATCH flow state uses Django cache (600s TTL), key: f'onboarding:{phone}'.
All outbound message strings are in templates.py as pure Python functions.

10. AI VERIFICATION CHAIN  (apps/incidents/tasks.py)
────────────────────────────────────────────────────────────────────────────────
Task: verify_incident_with_ai(incident_id)
  1. Fetches incident + media URLs
  2. Calls Anthropic claude-sonnet-4-6 with structured prompt
  3. Parses JSON response: { severity, fraud_score, confidence, recommendation }
  4. Updates incident.severity, ai_confidence, fraud_score, ai_raw_response
  5a. fraud_score > 0.7  → auto-reject + WhatsApp notify reporter
  5b. otherwise          → set VERIFIED + notify reporter + trigger dispatch

Celery dispatch chain (all async tasks):
  verify_incident_with_ai
    → notify_nearest_responders   (10km radius, 5 closest VERIFIED+available)
    → notify_nearest_organisations (10km radius, 5 closest VERIFIED)
    → notify_location_subscribers  (broad 10km sweep, per-sub radius filter)

Re-dispatch: notify_nearest_responders(incident_id, exclude_ids=[...]) — declined
  responders are passed in exclude_ids and skipped in the Haversine query.

────────────────────────────────────────────────────────────────────────────────
11. DONATIONS  (apps/resources/)
────────────────────────────────────────────────────────────────────────────────
Flow:
  1. POST /donate/initiate/
       Body: { fund_choice, donor_name, donor_email, amount_naira }
       → Paystack Initialize API
       → 10% transaction_charge split to SIREN_PAYSTACK_SUBACCOUNT
       → Returns { authorization_url } — frontend redirects user here

  2. User completes payment on Paystack hosted page
       → Paystack redirects to SITE_URL/donate/success?reference=...

  3. POST /donate/verify/  (Paystack sends this automatically)
       → Validate X-Paystack-Signature (HMAC SHA-512)
       → Idempotent: skip if Donation.status == SUCCESS already
       → Set status=SUCCESS, update incident totals, send thank-you WhatsApp

Fund choices: VICTIM | RESPONDER | PLATFORM

14. HAVERSINE UTILITY  (utils/distance.py)
────────────────────────────────────────────────────────────────────────────────
haversine_query(center_lat, center_lng, table, extra_filters='', radius_km=10.0, limit=5)

Executes raw SQL using the spherical law of cosines formula.
Uses HAVING (not WHERE) because the distance expression is an aggregate alias —
using WHERE on it raises a SQL column reference error in PostgreSQL.
